## Context
Use this when Chaos.Mongo needs a real benchmark instead of a synthetic microbenchmark. The goal is to exercise actual package wiring while keeping one-time environment setup out of the measured path.

## Patterns
- Put repository benchmarks in a dedicated `benchmarks/<ProjectName>/` executable project and add it to `Chaos.Mongo.slnx`.
- Reference the real source project under test instead of copying logic into the benchmark.
- Start a MongoDB 8 replica-set Testcontainer in `[GlobalSetup]`.
- Recreate service providers and unique database names in `[IterationSetup]` so each benchmark iteration runs against isolated collections without paying for container startup in the measurement.
- Warm both benchmark variants during setup when one path has one-time capability detection or collection creation work that should not skew steady-state results.
- When only a small set of benchmark parameter combinations is valid, model them as explicit scenario objects via `ParamsSource` instead of separate `[Params]` dimensions that can silently measure the wrong matrix.
- If BenchmarkDotNet's autogenerated project inherits repo-wide multitargeting, add a `benchmarks/Directory.Build.props` override that clears `TargetFrameworks` and pins `TargetFramework` to `$(LatestTarget)`.
- Prefer synchronous BenchmarkDotNet setup/cleanup hooks that block on async Mongo/Testcontainers calls with `GetAwaiter().GetResult()` when the BenchmarkDotNet version in use rejects async iteration hooks.

## Examples
- EventStore benchmark project: `benchmarks/Chaos.Mongo.EventStore.Benchmarks/`
- Single-target override for autogenerated BenchmarkDotNet builds: `benchmarks/Directory.Build.props`

## Anti-Patterns
- Benchmarking ad hoc helper methods instead of the repository's real DI-wired implementation.
- Measuring Docker container startup or first-time configurator/index creation as part of the append-path benchmark.
- Letting benchmark projects inherit repo-wide multitargeting when BenchmarkDotNet needs a single concrete runtime.
- Introducing static caches keyed by per-iteration `IMongoClient` instances; they retain benchmark-created clients across iterations and distort `MemoryDiagnoser` results.
